# Ableton UI Teardown: Echo

## Metadata

- Reference device: `Echo`
- Date: 2026-03-15
- Reviewer: Codex
- Sprint week: later reference set
- Why this device is in scope: benchmark for creative effect depth, spatial
  feedback readability, and multi-mode UI that stays calm

## Product Role

- Creative effect with rich internal behavior
- The product role is obvious because delay character, timing, and feedback
  remain central despite the extra modes and coloration options

## Hero Surface

- The main delay or feedback interaction zone is the hero surface
- It earns that role because the user is shaping time, space, and character at
  once
- Direct edits revolve around time, feedback, tone, and spatial behavior

## Always Visible Vs Contextual

- Always visible: the core time-and-feedback story
- Contextual: deeper color, modulation, or routing details
- Repo translation: creative effect lanes should expose the main behavior first
  and let deeper character controls remain secondary

## Selection Model

- Selection matters less than keeping the active time-and-feedback model clear
- Repo translation: when a device does not use selected objects, it still needs
  a clear central behavior model rather than a field of equal controls

## Compact Legibility

- The device remains identifiable because the main delay behavior is not buried
- Repo translation: future delay or spatial lanes should protect their compact
  identity even as character features expand

## State Encoding

- Active modes, feedback risk, and spatial behavior need distinct signals
- Repo translation: warning or performance-relevant states should be obvious
  without turning the UI into a hazard dashboard

## Motion, Analyzer, And Meter Behavior

- Motion should confirm time-based behavior without overpowering the controls
- Repo translation: animated delay or spatial visuals should explain the
  musical effect, not advertise themselves

## What To Steal

- Pattern to adopt in this repo: rich creative depth under a stable main
  behavior story
- Pattern to adapt rather than copy: calm mode layering for future delay or
  modulation-heavy effect lanes
- Pattern to avoid: exposing every creative branch at the same visual weight

## Repo Translation

- Target repo lane: future delay, echo, or modulation-rich effect lanes
- What should change first in that lane: define one obvious timing-and-feedback
  center
- What should stay different because the product role is different: not every
  repo lane needs Echo-level depth or color-stage complexity
